# Simulation

Using the OrefSwiftCLI command line utility, we should be able to
simulate the algorithm running. See the @OrefSwiftCLI.md file for more
details, but to simulate the algorithm we need to use these
subcommands from the OrefSwiftCLI:

  - initialize: Starts a new simulator session for a given virtual
    user. Requires a `--virtual-user` (`-u`) flag with the path to a
    virtual user directory (e.g., `VirtualUsers/sam`). The command
    loads the user's therapy settings files from that directory to
    create the initial simulation state.

  - stepUpdate: Updates the state of the algorithm each simulation
    step

  - calculate: Calculates insulin dosing for a given algorithm state
    and new glucose reading

In the iOS implementation you'll see mention of `simulation` but this
is a different simulation and can be ignored. The iOS simulation is
used to calculate the impact of carbs / insulin / etc on the dosing
calculator view.

## State directory

The `initialize` command automatically creates a state directory and
returns its path in the output. The directory is named using the
virtual user name and a timestamp (e.g., `state/sam_20260212_143022/`).
The `stepUpdate` and `calculate` subcommands require a `--state-dir`
(`-s`) flag pointing to the state directory returned by `initialize`.
State files are persisted between calls in this directory, making it
easy to inspect state between calls for debugging.

## Internal state

To enable simulations, the OrefSwiftCLI needs to manage internal state
to keep track of it's own internal view of the algorithm. All state
files live in the state directory.

### State created by `initialize`

The `initialize` command copies the virtual user's therapy settings
files into the state directory and then uses them to build the
initial state:

  - preferences.json, settings.json, bg_targets.json,
    basal_profile.json, insulin_sensitivities.json, carb_ratios.json,
    temptargets.json: Copied directly from the virtual user directory.
    Stored as separate files to match the iOS implementation. These
    are used by `calculate` to re-run makeProfile each cycle.

  - glucose.json: Empty array, populated by `calculate`.

  - pump_history.json: Empty array, populated by `calculate`.

  - carbs.json: Empty array, populated by `stepUpdate`.

  - autosens.json: Default autosens with ratio 1.0 and no timestamp,
    so it gets recalculated on the first `calculate` call once enough
    glucose data is available.

  - profile.json: The initial profile generated by running makeProfile
    with the provided settings.

### State updated by `calculate`

  - glucose.json: New glucose reading appended to the history.

  - pump_history.json: New pump events appended based on the
    determination result (temp basals, SMBs).

  - autosens.json: Recalculated every 30 minutes of simulated time.
    Until there are enough glucose readings (~72, about 6 hours of
    data), autosens keeps the default ratio of 1.0.

  - profile.json: Regenerated each cycle by running makeProfile.

  - determinations/: Directory containing the raw JSON output from
    determineBasal for each timestep. Each file is named with the
    simulated timestamp in local timezone ISO8601 format, e.g.,
    `determinations/2026-01-17T13:35:00-08:00.json`. The timestamp
    comes directly from the input argument passed to `calculate`.
    This is useful for debugging and analyzing algorithm behavior
    over a simulation run.

Note: There is a reservior in the iOS implementation but we can ignore
it since it's not used by the algorithm and just hard-code it to "1".

### State updated by `stepUpdate`

Currently, I'm unsure how the `stepUpdate` function is going to be
used by the broader simulation framework, so we'll just leave it as a
no-op for now.

### Initial state

In order for the simulation to run, we need to create some initial
states based on virtual users. The `initialize` command takes a
`--virtual-user` (`-u`) flag pointing to a virtual user directory
and loads the therapy settings files from it to build the
MakeProfileInputs struct used to create `settings.json`. This struct
contains all of the therapy settings that define how the algorithm
behaves for that user.

A virtual user directory must have the following json files,
typically copied from a Trio install that exposes these via the local
file system on a device. These should all have proper Swift types for
easy loading during simulation.

  - preferences.json: Algorithm tuning parameters.

  - settings.json: Pump configuration.

  - bg_targets.json: Blood glucose target ranges.

  - basal_profile.json: Array of basal rate entries throughout the day.

  - insulin_sensitivities.json: Insulin sensitivity factor schedule.

  - carb_ratios.json: Carb ratio schedule.

  - temptargets.json: Array of temporary BG targets.

  - model (not present, hard code): Pump model string (e.g., "722").

#### TrioCustomOrefVariables

The `calculate` command also needs a TrioCustomOrefVariables struct
when calling determineBasal. This is not part of the stored settings
but must be constructed at runtime. For simulation, use the same
algorithm that the iOS implementation uses to calculate the values of
this struct.

## Storage interfaces

To support storing and looking up data. The "fetch" functions return
data types that the algorithm expects. All `at` parameters represent
simulated time (not wall-clock time), so lookback windows are relative
to the simulated timestamp.

### Data retention and pruning

As an optimization, each `store*` method prunes records older than the
maximum lookback window for that data type. This keeps JSON file sizes
bounded so that read-modify-write cost stays constant as the
simulation progresses. The current retention windows are:

  - glucose.json: 24 hours (matches fetchGlucose lookback)
  - pump_history.json: 24 hours (matches fetchPumpEvents / calculateTDD)
  - carbs.json: 24 hours (matches fetchCarbs lookback)
  - tdd.json: 10 days (matches fetchTDDRecords default lookback)

These retention constants are defined in `SimulationStorage.swift`. If
any algorithm query is changed to look back further than the current
retention window, the corresponding constant MUST be updated to match
or the query will silently miss older data.

### Glucose

  - storeGlucose(at: Date, glucose: Decimal)

  // Fetches up to 24 hours of glucose before `at`, limited by `limit`.
  // The iOS code uses limit=72 for the full algorithm run and limit=6
  // for glucose validation checks.
  - fetchGlucose(at: Date, limit: Int?) -> [BloodGlucose]

#### Synthesizing BloodGlucose objects

`storeGlucose` only accepts a timestamp and value, but `BloodGlucose`
has many fields. The store must synthesize full objects on fetch. Use
these defaults:

  - sgv: Set from the stored glucose value (as Int)
  - glucose: Same as sgv
  - date: Milliseconds since epoch (as Decimal) from the stored
    timestamp
  - dateString: The stored timestamp as a Date
  - direction: Compute from consecutive readings if possible (delta
    between current and previous reading mapped to a trend arrow),
    otherwise default to .flat
  - noise: 0 (clean signal)
  - _id: A generated unique string (e.g., UUID)
  - All other fields (unfiltered, filtered, type, activationDate,
    sessionStartDate, transmitterID): nil

#### Direction calculation

The direction field affects GlucoseStatus in the algorithm. A simple
mapping from the 5-minute delta (current sgv - previous sgv):

  - delta > 17: .tripleUp
  - delta > 9: .doubleUp
  - delta > 4: .singleUp
  - delta > 2: .fortyFiveUp
  - delta >= -2: .flat
  - delta >= -4: .fortyFiveDown
  - delta >= -9: .singleDown
  - delta >= -17: .doubleDown
  - delta < -17: .tripleDown

If there is no previous reading, use .flat.

### Pump events

  // Stores a temp basal event. Rate is in units/hour and duration is
  // in minutes.
  - storeTempBasal(at: Date, rate: Decimal, duration: Int)

  // Stores an SMB (super micro bolus) event. Amount is in units.
  - storeSMB(at: Date, amount: Decimal)

  // Fetches 24 hours of pump events before `at`. Each stored temp
  // basal must be expanded into TWO PumpHistoryEvent entries to match
  // the format the algorithm expects:
  //   1. A TempBasal event (type: .tempBasal, rate, temp: .absolute)
  //   2. A TempBasalDuration event (type: .tempBasalDuration,
  //      durationMin set to the duration)
  // Both events share the same timestamp. The TempBasal event id
  // should be prefixed with "_" (e.g., "_<uuid>") while the
  // TempBasalDuration event uses the base id, matching the iOS
  // PumpEventDTO convention.
  //
  // Bolus/SMB events are single PumpHistoryEvent entries with
  // type: .bolus (or .smb), amount set, and isSMB: true for SMBs.
  - fetchPumpEvents(at: Date) -> [PumpHistoryEvent]

  // Looks back 20 minutes from `at` for the most recent temp basal
  // event. If found, subtracts the elapsed time from the original
  // duration to get remaining duration. If no temp basal is found
  // (or the remaining duration is <= 0), returns:
  //   TempBasal(duration: 0, rate: 0, temp: .absolute, timestamp: at)
  - fetchCurrentTempBasal(at: Date) -> TempBasal

### Carbs

  // Stores a carb entry at the given simulated time.
  - storeCarbs(at: Date, carbs: Decimal)

  // Fetches 24 hours of carb entries before `at`. Returns CarbsEntry
  // objects. Synthesize with defaults:
  //   - id: Generated UUID string
  //   - createdAt: The stored timestamp
  //   - actualDate: Same as createdAt
  //   - carbs: The stored value
  //   - fat, protein: nil
  //   - note, enteredBy: nil
  //   - isFPU: false
  //   - fpuID: nil
  - fetchCarbs(at: Date) -> [CarbsEntry]

## Mapping Determination results to stored pump events

After `calculate` runs determineBasal and gets a Determination, the
result must be stored back as pump events so subsequent algorithm
cycles see them. The relevant fields from Determination are:

  - rate: The temp basal rate (units/hour). If present, store a temp
    basal event using storeTempBasal with this rate and the duration
    from the determination.

  - duration: Temp basal duration in minutes. Used with rate above.

  - units: The SMB amount (units of insulin). If present and > 0,
    store an SMB event using storeSMB.

If the determination has no rate (nil), it means the algorithm is not
recommending a change to the current temp basal, so no temp basal
event should be stored.

## TempBasal commands semantics and the impact on TDD calculations

TempBasal commands run on real pumps are both mutually exclusive and
quantized.

In AID systems, at most one TempBasal command can run on the pump at
any given time. However, the system will issue a sequence of
overlapping TempBasal commands as a part of the natural running of the
system. For example, you might see a TempBasal command with rate: 1.5
and duration: 30m followed by another command with a rate: 1 and
duration: 2h five minutes later. To handle these cases correctly, the
simulation needs to process TempBasal commands in the TDD calculation
to truncate the duration accordingly when calculating insulin dosing.

In real pumps insulin delivery is quantized. In the simulation we will
ignore this quantization but we need to have a clear comment in the
TDD calculation to document this implementation decision.

Finally, it's important that we store the raw TempBasal command issued
to the pump with the full duration because other parts of the
algorithm use this information to act, particularly when reasoning
about the currently running TempBasal command. Thus, these
modifications are isolated to the TDD calculation.

## Algorithm execution order in calculate

The `calculate` command mirrors the iOS APSManager.determineBasal
flow. For reference, the execution order is:

  1. Store the new glucose reading (storeGlucose)
  2. Run makeProfile using stored settings to regenerate the profile
  3. Fetch glucose, pump history, and carbs from storage
  4. Check autosens cache: if the stored autosens timestamp is more
     than 30 minutes old (in simulated time), recalculate it.
     Autosens needs ~72 glucose readings (~6 hours) to produce a
     meaningful result; until then, keep the default ratio of 1.0.
  5. Run iob(history: pumpEvents, profile: profile, clock: now,
     autosens: autosens)
  6. Run meal(pumpHistory: pumpEvents, profile: profile,
     basalProfile: basalProfile, clock: now, carbs: carbs,
     glucose: glucose)
  7. Run determineBasal(glucose: glucose, currentTemp: currentTemp,
     iob: iob, profile: profile, autosens: autosens, meal: meal,
     microBolusAllowed: true, reservoir: 1,
     pumpHistory: pumpEvents, preferences: preferences,
     basalProfile: basalProfile,
     trioCustomOrefVariables: trioCustomOrefVariables)
  8. Store the resulting pump events from the determination
